# 메이크로컬 CODEX_RULES.md v0.1

## 1. 문서 목적

이 문서는 메이크로컬 MVP를 Codex와 함께 개발할 때 반드시 지켜야 할 작업 규칙을 정의한다.

목적은 다음과 같다.

1. 요청하지 않은 파일이 수정되는 것을 방지한다.
2. 프로젝트 구조가 개발 중에 흐트러지는 것을 방지한다.
3. 기능별로 작게 개발하고 검증할 수 있게 한다.
4. 비개발자인 창업자가 Codex 작업 결과를 추적하고 통제할 수 있게 한다.
5. 추후 유지보수와 확장이 가능한 코드베이스를 만든다.

Codex는 모든 개발 작업에서 이 문서를 우선적으로 준수해야 한다.

## 2. 최상위 원칙

### 2.1 작은 단위로 작업한다

한 번에 여러 기능을 만들지 않는다.

예를 들어 다음과 같은 요청은 금지한다.

```txt
메이크로컬 플랫폼 전체를 만들어줘.
```

다음과 같이 작은 기능 단위로 작업해야 한다.

```txt
제작 의뢰 1단계 파일 업로드 페이지를 구현해줘.
```

### 2.2 요청받은 범위 밖의 파일을 수정하지 않는다

Codex는 작업 지시서에 명시된 `수정 허용 파일`만 수정한다.

수정이 필요한 파일이 추가로 발견되면 즉시 작업을 멈추고, 왜 추가 수정이 필요한지 설명한 뒤 승인을 받아야 한다.

### 2.3 기존 동작을 임의로 바꾸지 않는다

새 기능을 추가하더라도 기존 기능의 동작을 바꾸면 안 된다.

기존 동작 변경이 필요한 경우 다음을 먼저 설명해야 한다.

- 어떤 기존 동작을 바꾸는지
- 왜 변경이 필요한지
- 변경으로 영향을 받는 페이지나 컴포넌트는 무엇인지
- 테스트 방법은 무엇인지

### 2.4 문서와 구조를 우선한다

기능을 추가하기 전에 관련 문서와 기존 구조를 확인한다.

기능 추가로 인해 다음 문서의 변경이 필요하다면, 코드 수정 전에 문서를 먼저 업데이트해야 한다.

- `docs/PRODUCT_SPEC.md`
- `docs/ARCHITECTURE.md`
- `docs/DB_SCHEMA.md`
- `docs/CODEX_RULES.md`
- `docs/TASK_LOG.md`

### 2.5 작업 후 반드시 보고한다

Codex는 작업 후 다음 항목을 반드시 보고한다.

1. 변경한 파일 목록
2. 주요 구현 내용
3. 실행 방법
4. 테스트 방법
5. 남은 이슈 또는 주의사항

## 3. 작업 지시서 형식

모든 Codex 작업 요청은 가능한 한 다음 형식을 따른다.

```txt
[프로젝트 배경]
메이크로컬은 시제품 제작 수요자와 지역 중소 제조업체를 연결하는 제조 매칭 MVP입니다.

[현재 목표]
이번 작업의 목표는 ______ 입니다.

[수정 허용 파일]
- 파일 경로 1
- 파일 경로 2

[수정 금지 파일]
- 파일 경로 또는 폴더 1
- 파일 경로 또는 폴더 2

[기능 요구사항]
- 요구사항 1
- 요구사항 2
- 요구사항 3

[제외 범위]
- 이번 작업에서 하지 않을 것 1
- 이번 작업에서 하지 않을 것 2

[기술 조건]
- Next.js App Router
- TypeScript
- Tailwind CSS
- Supabase 사용

[완료 기준]
- 기준 1
- 기준 2
- 기준 3

[작업 후 보고]
- 변경 파일 목록
- 주요 구현 내용
- 실행 방법
- 테스트 방법
- 남은 이슈
```

## 4. 수정 허용 파일과 수정 금지 파일 규칙

### 4.1 수정 허용 파일

작업 지시서에 명시된 파일만 수정한다.

예시:

```txt
[수정 허용 파일]
- app/request/upload/page.tsx
- components/request/FileUploadStep.tsx
- lib/storage/uploadRequestFile.ts
- types/request.ts
```

### 4.2 수정 금지 파일

작업 지시서에 명시된 금지 파일이나 폴더는 수정하지 않는다.

예시:

```txt
[수정 금지 파일]
- app/page.tsx
- app/admin/**
- components/landing/**
- database/schema.sql
```

### 4.3 예외 처리

작업 중 수정 금지 파일을 수정해야만 기능이 동작한다고 판단되면, Codex는 즉시 멈추고 다음 형식으로 보고한다.

```txt
추가 수정이 필요한 파일:
- 파일 경로

필요한 이유:
- 이유 설명

영향 범위:
- 영향을 받는 기능 또는 페이지

승인 후 진행하겠습니다.
```

## 5. 프로젝트 구조 유지 규칙

### 5.1 라우트 파일 규칙

`app/` 폴더의 `page.tsx`는 라우팅과 화면 조립 역할을 한다.

긴 UI나 복잡한 로직을 `page.tsx`에 직접 작성하지 않는다.

### 5.2 컴포넌트 분리 규칙

UI는 기능에 따라 `components/` 폴더에 분리한다.

- 랜딩페이지: `components/landing/`
- 제작 의뢰 사용자 흐름: `components/request/`
- 관리자 화면: `components/admin/`
- 제조업체 관련 UI: `components/suppliers/`
- 공통 UI: `components/common/` 또는 `components/ui/`

### 5.3 비즈니스 로직 분리 규칙

비즈니스 로직은 페이지 컴포넌트에 직접 작성하지 않는다.

- Supabase 관련 로직: `lib/supabase/`
- AI 관련 로직: `lib/ai/`
- 파일 업로드 로직: `lib/storage/`
- 업체 매칭 로직: `lib/matching/`
- 검증 로직: `lib/validators/`

### 5.4 타입 정의 규칙

TypeScript 타입은 `types/` 폴더에 정의한다.

예시:

- `types/request.ts`
- `types/supplier.ts`
- `types/rfq.ts`
- `types/recommendation.ts`
- `types/common.ts`

동일한 타입을 여러 파일에 중복 정의하지 않는다.

## 6. 데이터베이스 작업 규칙

### 6.1 DB 구조 변경 금지 원칙

Codex는 작업 지시 없이 DB 테이블명, 컬럼명, 타입을 임의로 변경하지 않는다.

DB 변경이 필요하다면 먼저 `docs/DB_SCHEMA.md`를 업데이트하고, 변경 이유를 설명해야 한다.

### 6.2 SQL 파일 위치

DB 스키마는 다음 파일에 저장한다.

```txt
database/schema.sql
```

초기 테스트용 데이터는 다음 파일에 저장한다.

```txt
database/seed.sql
```

### 6.3 Supabase 접근 규칙

Supabase 쿼리는 페이지 컴포넌트에 직접 작성하지 않는다.

다음 위치에 함수로 분리한다.

```txt
lib/supabase/requests.ts
lib/supabase/suppliers.ts
lib/supabase/files.ts
lib/supabase/recommendations.ts
```

### 6.4 RLS 및 보안

- 클라이언트에서 서비스 역할 키를 사용하지 않는다.
- `SUPABASE_SERVICE_ROLE_KEY`는 서버 API 라우트에서만 사용한다.
- 공개 사용자가 직접 DB에 쓰기 권한을 갖지 않도록 한다.
- 로그인 기능 도입 전까지는 서버 API를 통해 저장한다.

## 7. 파일 업로드 작업 규칙

### 7.1 허용 파일 형식

초기 MVP에서 허용하는 파일 형식은 다음과 같다.

- PDF
- STEP
- STP
- STL
- JPG
- JPEG
- PNG

ZIP 파일은 허용하지 않는다.

### 7.2 파일 크기 제한

초기 파일 크기 제한은 20MB로 설정한다.

### 7.3 파일 없는 진행 허용

도면이나 파일이 없어도 제작 의뢰를 진행할 수 있어야 한다.

이를 위해 파일 업로드 페이지에는 다음 버튼을 제공한다.

- 파일 업로드하고 다음 단계로
- 도면 없이 의뢰서 작성하기

### 7.4 Storage 경로

Supabase Storage 경로는 다음 규칙을 따른다.

```txt
requests/{requestId}/{timestamp}_{safeFileName}
```

## 8. AI 기능 작업 규칙

### 8.1 AI 호출 위치

AI API는 클라이언트 컴포넌트에서 직접 호출하지 않는다.

AI 호출은 다음 API 라우트를 통해 수행한다.

```txt
app/api/ai/generate-rfq/route.ts
```

실제 프롬프트와 응답 파싱 로직은 다음 위치에 둔다.

```txt
lib/ai/generateRfq.ts
lib/ai/prompts.ts
```

### 8.2 AI가 생성할 수 있는 것

AI는 다음 항목을 생성한다.

- 제작의뢰서 초안
- 추천 제작방법
- 예상 견적 범위
- 예상 납기 일정
- 누락 정보 체크리스트
- 업체 확인 질문 리스트
- 견적 상승 요인

### 8.3 AI가 하면 안 되는 것

AI는 다음을 확정적으로 단정하면 안 된다.

- 최종 견적 확정
- 최종 제작 가능 여부 확정
- 최종 납기 확정
- 법적 또는 품질 보증성 문구
- 실제 제조업체의 확정 답변처럼 보이는 표현

### 8.4 AI 표현 원칙

다음 표현을 사용한다.

- 예상 견적 범위
- 예상 납기 일정
- 업체 검토 후 확정
- 확인 필요
- 추가 정보 필요

다음 표현은 피한다.

- 확정 견적
- 반드시 제작 가능
- 확정 납기
- 보장

### 8.5 AI 응답 구조

AI 응답은 가능하면 JSON 구조로 받는다. UI에 바로 자연어를 붙여넣기보다, JSON을 파싱해서 화면 항목별로 표시한다.

## 9. UI 작업 규칙

### 9.1 디자인 원칙

- 전문적이고 신뢰감 있는 B2B 톤을 유지한다.
- 제조업체와 스타트업 모두 이해할 수 있는 쉬운 용어를 사용한다.
- 과도한 애니메이션이나 장식은 피한다.
- 입력 화면은 단계별로 단순하게 구성한다.
- 사용자가 현재 몇 단계인지 알 수 있도록 진행 단계 표시를 제공한다.

### 9.2 버튼 문구 원칙

주요 버튼 문구는 명확하게 작성한다.

예시:

- 제작 의뢰 시작하기
- 파일 업로드하고 다음 단계로
- 도면 없이 의뢰서 작성하기
- 제작의뢰서 생성하기
- 추천 업체 확인하기
- 운영자에게 맡기기
- 제작 문의 제출하기

### 9.3 AI 결과 표시 원칙

AI가 채운 항목은 사용자가 수정할 수 있어야 한다.

AI 추천 항목에는 다음과 같은 안내를 표시한다.

```txt
AI가 입력 정보와 첨부 파일을 바탕으로 추정한 내용입니다. 실제 견적과 제작 가능 여부는 제조업체 검토 후 확정됩니다.
```

## 10. 관리자 화면 작업 규칙

### 10.1 관리자 기능 우선순위

관리자 화면은 초기 MVP 운영에 매우 중요하다.

우선순위는 다음과 같다.

1. 제작 의뢰 목록 확인
2. 제작 의뢰 상세 확인
3. AI 의뢰서 및 사용자 최종 의뢰서 확인
4. 첨부 파일 확인
5. 제조업체 DB 확인
6. 추천 업체 확인 및 수정
7. 상태 변경
8. 운영자 메모

### 10.2 관리자 인증

초기 개발 중에는 임시 접근 방식이 가능하지만, 배포 시에는 최소한의 보호가 필요하다.

권장 초기 방식:

- 환경변수 `ADMIN_ACCESS_TOKEN` 기반 접근 제한

정식 서비스 전환 시:

- Supabase Auth 기반 관리자 로그인 도입

## 11. 매칭 기능 작업 규칙

### 11.1 초기 매칭 방식

초기에는 복잡한 AI 매칭이 아니라 룰 기반 매칭을 사용한다.

추천 기준:

- 공정 일치
- 지역 일치
- 소재 대응 가능 여부
- 소량 시제품 대응 가능 여부
- 후처리 가능 여부
- 관리자 메모

### 11.2 추천 사유

추천 결과에는 반드시 추천 사유를 표시한다.

예시:

```txt
알루미늄 소량 MCT 가공 대응 가능, 대구권 업체, 시제품 제작 경험 보유
```

### 11.3 운영자에게 맡기기

사용자는 추천 업체를 직접 선택하지 않고 “운영자에게 맡기기”를 선택할 수 있어야 한다.

이 경우 `supplier_recommendations`에 `is_operator_delegated = true`로 저장한다.

## 12. 테스트 규칙

### 12.1 작업별 테스트 체크리스트 작성

각 기능 작업 후에는 간단한 테스트 체크리스트를 작성한다.

예시: 파일 업로드 기능

```txt
- PDF 업로드 성공
- STEP 업로드 성공
- STL 업로드 성공
- JPG 업로드 성공
- PNG 업로드 성공
- ZIP 업로드 거부
- 파일 없이 다음 단계 이동 가능
- 20MB 초과 파일 업로드 거부
- 오류 메시지 표시
```

### 12.2 기본 테스트 항목

모든 작업 후 다음을 확인한다.

- 로컬 실행 오류 없음
- TypeScript 오류 없음
- 기존 페이지 접근 가능
- 수정 대상 기능 정상 동작
- 수정하지 않은 기능이 깨지지 않음

## 13. Git 작업 규칙

### 13.1 브랜치

- `main` : 안정 버전
- `dev` : 개발 버전

초기에는 `dev`에서 작업하고, 안정화 후 `main`에 반영한다.

### 13.2 커밋 단위

하나의 기능 단위로 커밋한다.

좋은 예시:

```txt
feat: add request file upload page
feat: add rfq generation api
feat: add supplier admin table
fix: restrict unsupported file uploads
docs: add database schema
```

나쁜 예시:

```txt
update stuff
fix everything
make app
```

### 13.3 작업 전후 커밋

Codex에게 큰 작업을 맡기기 전에는 현재 상태를 커밋한다.

작업 후 정상 동작을 확인하면 다시 커밋한다.

문제가 생기면 마지막 정상 커밋으로 되돌린다.

```txt
- 작업 목표

- 파일 1

- 구현 내용 1

- 테스트 항목 2: 결과

- 이슈 1
- 이슈 2
```

## 15. 금지 사항

Codex는 다음 행동을 하면 안 된다.

1. 요청하지 않은 파일 대량 수정
2. 기존 폴더 구조 임의 변경
3. 기존 타입 중복 정의

## 16. Codex 작업 완료 보고 템플릿

Codex는 작업을 완료한 뒤 다음 형식으로 보고한다.

```txt
## 작업 완료 보고

### 변경 파일
- 파일 경로 1
- 파일 경로 2

### 주요 구현 내용
- 내용 1
- 내용 2

### 실행 방법
- 명령어 또는 접근 경로

### 테스트 방법
- 테스트 1
- 테스트 2

### 확인 필요 사항
- 확인할 점 1
- 확인할 점 2

### 남은 이슈
- 이슈 1
- 이슈 2
```

## 17. 첫 번째 개발 작업 전 준비사항

Codex에게 첫 개발 작업을 맡기기 전에 다음 문서가 존재해야 한다.

- `docs/PRODUCT_SPEC.md`
- `docs/ARCHITECTURE.md`
- `docs/DB_SCHEMA.md`
- `docs/CODEX_RULES.md`
- `docs/TASK_LOG.md`

첫 개발 작업은 다음 범위로 제한한다.

```txt
Next.js + TypeScript + Tailwind 기반 프로젝트 초기 세팅
기본 폴더 구조 생성
문서 파일 생성
랜딩페이지 임시 뼈대 생성
Supabase 클라이언트 설정 파일 생성
```

첫 작업에서는 다음 기능을 구현하지 않는다.

- 제작 의뢰 폼
- 파일 업로드
- AI API 호출
- 관리자 대시보드
- 제조업체 DB
- 업체 추천

## 18. 이 문서의 우선순위

Codex 작업 중 판단이 어려운 경우 다음 우선순위를 따른다.

1. 사용자가 명시한 작업 지시
2. `docs/CODEX_RULES.md`
3. `docs/ARCHITECTURE.md`
4. `docs/DB_SCHEMA.md`
5. `docs/PRODUCT_SPEC.md`
6. 기존 코드 구조

위 기준이 충돌할 경우 작업을 중단하고 사용자에게 확인을 요청한다.
